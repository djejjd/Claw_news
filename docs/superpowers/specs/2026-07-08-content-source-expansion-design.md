# 内容源扩充与三类正式发布设计

## 背景

当前项目的展示层已经调整为单条摘要消息，并按 `AI / 工具 / 游戏` 三组输出。但正式 ingest 与 publish 主链路仍然偏向 AI 内容：

- `run_ingest()` 只接入 `AI RSS + HuggingFace + GitHub`
- `RunContext.publish_scope` 仍固定为 `ai_only`
- 历史 RSS 分类中仍存在 `device`

这会导致展示结构已经变更，但内容层无法稳定产出三类内容，尤其是 `工具` 与 `游戏` 组来源不足。

## 目标

在不改动当前消息展示样式的前提下，补齐正式内容链路，使项目能够稳定生成以中文为主、兼顾少量高质量英文内容的 `AI / 工具 / 游戏` 单条摘要。

## 非目标

本轮不做以下事项：

- 不重做评分体系
- 不加入反馈体系
- 不修改当前单条 10 项摘要样式
- 不为三类内容拆成三套独立 pipeline
- 不引入高维护成本、强反爬依赖的新爬虫体系

## 用户确认的范围

- 内容优先级：`AI + 工具 + 游戏` 三类同时补齐
- 语言偏好：中文优先，辅以少量高信噪比英文源
- 推送形态：继续保持单条消息、最多 10 条、按 `AI / 工具 / 游戏` 分组
- 本轮目标：先补内容，再考虑展示优化

## 现状与冲突点

### 1. 发布范围与展示范围不一致

当前 `app/pipeline/context.py` 中：

- `PublishScope = Literal["ai_only"]`
- `RunContext.publish_scope` 默认值为 `ai_only`

这与现有展示层的三组结构不一致。如果不调整，`工具` 与 `游戏` 只能靠局部映射或偶发内容出现，无法成为正式支持范围。

### 2. 分类口径不一致

当前仓库中仍有如下历史分类：

- `ai`
- `game`
- `device`

而展示层已经收口到：

- `AI`
- `工具`
- `游戏`

因此本轮必须明确：`device` 不再作为独立发布分类存在，而是统一映射到 `工具`。

### 3. 采集器存在，但正式链路未纳入

仓库已有以下能力：

- RSS 采集器
- TapTap 采集器
- 游戏 RSS 源（如游研社）
- GitHub 补充项目流

但正式 ingest 主链路当前只接入 AI 相关采集器。问题不在于“没有采集器”，而在于“正式链路没有把三类内容纳入同一套发布约束”。

## 设计原则

1. 复用现有统一 pipeline，不拆新系统
2. 优先接入 RSS 或稳定页面源，降低维护成本
3. 中文优先，英文只保留官方或高质量补充源
4. 先统一分类口径，再扩源
5. 单源失败不影响整轮 ingest / publish

## 方案对比

### 方案 A：只扩 RSS 源，保留 `ai_only` 发布

优点：

- 改动最小
- 接入速度快

缺点：

- 与现有 `AI / 工具 / 游戏` 展示目标不一致
- 内容层仍会长期偏 AI

### 方案 B：扩 RSS 源，并将正式 ingest / publish 放宽到三类

优点：

- 与当前展示结构一致
- 改动集中在配置、ingest、publish 三层
- 能最快形成“内容层与展示层统一”

缺点：

- 需要显式收口旧分类 `device`
- 需要补三类链路测试

### 方案 C：为三类内容拆独立 pipeline

优点：

- 长期边界清晰

缺点：

- 明显超出本轮范围
- 实现与维护成本高

## 结论

采用 **方案 B**：

在现有统一 pipeline 内扩充三类来源，并将正式发布范围从 `ai_only` 扩展为面向 `AI / 工具 / 游戏` 的统一发布口径。

## 首批接入内容源

### AI

- 保留：`qbitai`
- 新增：机器之心
- 新增：AI 产品/应用动态类中文源（如 APPSO / 爱范儿 AI 相关栏目，按可维护性择一）
- 保留可选英文：`huggingface`
- 英文官方源作为二批：`openai_blog`、`anthropic`

### 工具

- 保留并改口径：`sspai` 从 `device` 迁移到 `tool`
- 保留并改口径：`ithome` 中适合作为工具资讯的 RSS 归入 `tool`
- 新增：InfoQ 中文
- 新增：腾讯云开发者 / 阿里云开发者社区中可稳定接入的一类源
- GitHub 项目流继续保留为补充内容

### 游戏

- 保留：`yystv`
- 保留：`taptap`
- 新增：GameLook
- 新增：触乐或其他具备稳定 RSS/页面结构的游戏资讯源

## 分类与发布口径

### 内部候选分类

本轮允许正式进入 ingest / publish 的候选分类为：

- `ai`
- `tool`
- `game`

### 历史分类收口规则

- `device -> tool`

### 展示分类映射

- `ai -> AI`
- `tool -> 工具`
- `game -> 游戏`

## 架构设计

### 1. 源配置层

将当前偏单类的 RSS 配置扩展为明确支持三类内容：

- AI RSS 配置
- 工具 RSS 配置
- 游戏 RSS 配置

配置方式保持与现有 `collectors/ai_rss.py` 类似的轻量模式：

- 提供默认内置源
- 支持通过环境变量追加或替换

不引入数据库或复杂配置中心。

### 2. 正式 ingest 层

`app/scheduler/jobs.py::run_ingest()` 将不再只面向 AI：

- RSS 采集阶段允许返回三类内容
- 游戏类的 TapTap 采集器进入正式主链路
- HuggingFace 继续作为 AI 可选源
- GitHub 保持项目补充流，不要求与 headline 主池完全等价

每个 source 保持独立异常隔离：

- 任一源失败，不中断整轮 ingest
- 状态继续写入 `IngestStatusStore`
- metrics/state 继续按 source 记录

### 3. publish 层

`app/pipeline/news_pipeline.py` 将从“只发布 AI”扩展为“发布三类正式候选”：

- 去除仅按 `ai_only` 过滤的主逻辑
- 统一在 pipeline 内对三类内容做分类、选材、摘要
- 最终仍输出单条 10 项摘要

### 4. 评分层

本轮不重做评分体系，只做最小必要兼容：

- 三类候选均可进入当前选材逻辑
- 旧 `device` 权重配置统一向 `tool` 对齐
- 若部分新源暂无专门权重，先使用同类默认权重

## 数据与接口影响

### RunContext

需要扩展 `publish_scope` 的定义，使其不再只能表达 `ai_only`。

推荐方向：

- 保留已有默认值兼容旧测试和旧入口
- 新增一个明确表示三类正式发布的 scope

本轮不要求支持任意自由组合 scope，只支持：

- 兼容旧 `ai_only`
- 新增统一三类 scope

### CandidateItem / SummaryItem

本轮不要求新增复杂字段，但要求：

- 候选分类能覆盖 `tool`
- 展示分类映射稳定输出 `AI / 工具 / 游戏`

## 错误处理

- 单源采集失败：记录失败并继续
- 可选源失败：记录为 skipped 或 optional failure，不阻断
- 某一类当轮无内容：摘要允许只输出有内容的组，不强行补空位
- 某新源摘要较短：沿用已降低的最小摘要阈值

## 测试策略

本轮至少补齐以下测试：

1. 新三类源配置可正确加载
2. `device -> tool` 分类映射正确
3. `run_ingest()` 可写入 `ai / tool / game` 三类候选
4. TapTap 正式进入 ingest 主链路后，不影响现有容错模型
5. publish 不再只能过滤 AI
6. 三类候选进入摘要时，能稳定落到 `AI / 工具 / 游戏`
7. 单源失败时，其余源仍继续运行

## 验收标准

满足以下条件即可认为本轮完成：

1. 正式链路能接收并发布 `AI / 工具 / 游戏` 三类内容
2. 首批中文源已接入并通过基础测试
3. `device` 不再作为正式展示分类存在
4. 推送样式保持当前单条 10 项结构不变
5. 单源失败不影响整体服务健康

## 实施边界

本轮只做“内容层补齐”：

- 做正式源接入
- 做分类收口
- 做主链路放宽

不做：

- 反馈体系
- 新评分模型
- 展示文案二次设计
- 后台管理或配置 UI
