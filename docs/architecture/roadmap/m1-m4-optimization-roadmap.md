# Claw_news 架构评审与优化路线图

## 1. 评审结论

当前项目已经具备清晰的业务主链路：

- `collectors/` 负责多源采集
- `aggregator/` 负责评分和选品
- `pusher/` 负责企业微信推送
- `main.py` 负责任务编排

这说明项目已经从“临时脚本”走到了“结构化脚本应用”的阶段，核心业务抽象是成立的。

但从工程化角度看，它仍然更接近“单机可运行工具”，还没有进入“可长期维护、可稳定部署、可安全演进的生产应用”阶段。主要原因不是业务逻辑本身，而是配置契约、状态管理、幂等性、部署标准化、可观测性和安全边界不足。

一句话判断：

> 业务结构基本合理，系统工程边界偏弱，适合继续迭代，但不适合直接按长期生产系统标准运行。

---

## 2. 当前架构评价

## 2.1 合理的部分

- 分层是清楚的，采集、聚合、推送三个主职责没有完全混在一起。
- `HotItem` 作为统一数据模型，降低了数据源之间的耦合。
- 聚合规则集中在 `aggregator/merger.py`，业务规则有单点入口。
- 已有测试覆盖评分、推送格式和部分容错行为，说明项目不是纯手工验证。
- 入口简单，部署门槛低，适合快速迭代业务规则。

## 2.2 不足的部分

- `main.py` 仍承担了过多系统职责：配置读取、日志初始化、采集编排、状态管理、推送事务控制、历史清理都集中在一个文件。
- collector 直接自己读 `config.yaml`，导致配置边界不清晰，模块并不真正独立。
- 运行状态使用本地 JSON 文件保存，适合单机场景，不适合多次重试、并发触发和长期演进。
- 调度依赖 `launchd`/`cron`，说明部署形态仍然是“机器驱动脚本”，不是“标准化应用交付”。

## 3. 按优先级排序的危险问题

## P0: 安全风险

### 问题

文档中出现了真实格式的企业微信 webhook。

### 为什么是问题

Webhook 本质上就是发送权限。一旦泄露，任何人都能向目标群发送内容。

### 后果

- 群消息被滥发
- 机器人被滥用
- 安全审计不可追踪
- 后续即使代码修复，历史泄露仍然存在

### 推荐修改

- 立即轮换 webhook
- 仓库和文档中只保留占位符
- 配置迁移到环境变量或 secret 管理
- 后续补仓库 secret 扫描

### 是否值得现在修改

值得，必须立即处理。

### 推荐替代方案

- `.env` + 环境变量注入
- 部署平台 secret
- 统一 `Settings` 加载层

---

## P0: 推送事务不完整，状态可能与外部结果不一致

### 问题

当前推送是按分类串行发送，但本地状态只在全部完成后统一写入。

### 为什么是问题

如果 AI 已推送成功、Game 推送成功、Device 推送失败，本地 `pushed_urls.json` 和日报记录不会保存，但外部已经产生了部分真实发送。

### 后果

- 重试时重复推送
- `[新]` / `[续]` 判定失真
- 日志和真实结果不一致
- 后续问题排查困难

### 推荐修改

- 每个分类独立记录发送结果
- 校验企业微信响应体中的业务状态，而不是只看 HTTP 200
- 引入幂等键，例如 `date + period + category`
- 逐步把状态从 JSON 迁到 SQLite

### 是否值得现在修改

非常值得，属于生产可用性核心问题。

### 推荐替代方案

- 文件锁 + 分类级提交
- SQLite `push_runs` / `push_items` 表

---

## P1: 配置设计与实现不一致

### 问题

`config.example.yaml` 中定义了 `rss_feeds`，但实际采集器使用的是代码内写死的 `FEED_CONFIGS`。

### 为什么是问题

表面上看是“可配置系统”，实际上是“半配置半硬编码系统”。

### 后果

- 修改配置不会生效
- 文档与实现长期漂移
- 扩展新源必须改代码
- 维护者会对系统行为产生错误预期

### 推荐修改

- 统一从配置读取 RSS feed 列表
- 配置在启动时一次性加载并校验
- collector 只接收配置对象，不自行打开文件

### 是否值得现在修改

值得，属于低成本高收益项。

### 推荐替代方案

- `Settings` 对象集中注入
- `pydantic` / dataclass schema 校验

---

## P1: 测试契约不稳定，依赖本地隐式环境

### 问题

系统 Python 下 `pytest -q` 无法直接跑通；仓库内 `venv` 中可以通过。

### 为什么是问题

这说明项目的“可运行”依赖本地环境偶然成立，不是可复现的工程契约。

### 后果

- 新机器接手成本高
- CI 难以接入
- 容易出现“我这里可以，你那里不行”

### 推荐修改

- 增加 `pyproject.toml`
- 区分 runtime / dev 依赖
- 显式加入 `pytest-asyncio`
- 提供统一命令：`make test`

### 是否值得现在修改

值得，应尽快补齐。

### 推荐替代方案

- Poetry
- uv
- PDM

---

## P1: 本地文件状态缺少并发保护

### 问题

`pushed_urls.json`、日报 JSON、日志文件都没有并发控制。

### 为什么是问题

只要出现重复触发、人工补跑、crontab 重叠执行，就可能出现覆盖或竞争。

### 后果

- 重复推送
- 状态损坏
- 日志交叉污染
- 历史记录不可信

### 推荐修改

- 引入任务锁
- 写文件时使用原子写入
- 中期迁移到 SQLite

### 是否值得现在修改

如果准备长期运行，值得尽快处理。

### 推荐替代方案

- `flock`
- `portalocker`
- SQLite 事务

## 4. 技术债务清单

## 4.1 已形成的技术债务

- 配置读取逻辑分散在多个模块
- `main.py` 职责过重
- 测试中复制了 `safe_collect` 逻辑，而不是测试真实实现
- README 中仍引用已废弃的 `ithome.py`
- 文档和代码演进不同步
- 依赖只有 `requirements.txt`，没有更严格的工程元数据
- 没有 CI
- 没有容器化交付
- 没有告警、没有健康检查、没有结构化日志

## 4.2 债务性质判断

- 这些问题大多不是“今天就会崩”的问题
- 但它们会显著放大未来每一次改动的成本
- 如果继续加新源、新推送渠道、新调度方式，债务会迅速累积

---

## 5. 明显反模式

## 5.1 伪配置化

配置文件存在，但关键数据源列表仍写死在代码中。这是典型“看起来可配置，实际上不可配置”的反模式。

## 5.2 脚本入口承担应用内核职责

`main.py` 当前既像 CLI，又像 service，又像状态协调器。这会导致未来任何新增需求都继续往入口堆。

## 5.3 测试影子实现

`tests/test_resilience.py` 中复制了一份 `safe_collect` 逻辑，这会让测试和真实代码产生双轨维护。

## 5.4 单机状态假设

当前系统默认假设：

- 永远只有一个实例运行
- 永远不会重复触发
- 永远不需要补偿重试

这是脚本系统常见假设，但不适合作为长期系统前提。

## 6. Docker / Compose 评价

当前仓库没有 `Dockerfile` 或 Compose 文件。

这本身不算错误，但它意味着项目目前没有标准化运行时封装。对于个人本地 `launchd` 场景尚可，但对以下目标不友好：

- Linux 服务器部署
- CI 验证
- 多环境一致性
- 团队接手
- 迁移到容器平台

### 结论

- 现状：可本地运行
- 不足：不可标准化交付
- 建议：补齐最小容器交付面

### 最小建议交付物

- `Dockerfile`
- `docker-compose.yml`
- `.env.example`
- `data/` 挂载
- `healthcheck`
- 明确时区和日志目录

## 7. 长期维护适配性评价

当前项目适合：

- 单人维护
- 单机定时运行
- 低频需求变更
- 小范围业务规则迭代

当前项目不适合直接承担：

- 多环境部署
- 多实例并发运行
- 多推送渠道
- 严格审计与告警
- 长期多人协作扩展

### 判断

如果目标是“我自己长期用、偶尔改”，当前架构还能承受。  
如果目标是“持续演进、加功能、上服务器、交给别人接手”，需要开始系统化重构。

## 8. 性能瓶颈分析

## 8.1 当前主要瓶颈

- 外部网络 I/O，不是本地 CPU
- RSS 采集是顺序阻塞
- HF 摘要翻译依赖免费翻译路径，稳定性弱
- 推送失败恢复能力差

## 8.2 是否存在明显性能灾难

目前没有明显吞吐灾难，因为总数据量很小。  
当前更大的问题其实不是“慢”，而是“不稳定”和“不可预测”。

## 8.3 后续可能放大的瓶颈

- 数据源数量增加
- 翻译量增加
- 增加 HTML 富化抓取
- 引入更多推送渠道

### 建议

- 为所有外部请求统一设置 timeout / retry / backoff
- 对采集耗时打点
- 逐步统一 HTTP client 管理
- 对翻译做缓存或降级策略

---

## 9. 安全风险分析

当前主要安全问题有四类：

### 9.1 Secret 暴露

最高优先级问题，前文已列。

### 9.2 外部输入完全信任

采集的标题、摘要、链接来自外部站点，当前只有简单 HTML 去标签，没有更严格的清洗与异常字符控制。

### 9.3 部署脚本默认真实推送

`deploy.sh` 最后直接执行真实发送，这对生产环境不够稳妥。

### 9.4 缺少最小权限和审计边界

当前没有发送审计、没有异常告警、没有 secret 生命周期管理。

### 建议

- 部署脚本默认 `--dry-run`
- secret 外置
- 输出日志避免泄露敏感 URL
- 增加失败告警通道

---

## 10. 未来扩展风险

如果未来要扩展，最容易卡住的是以下几点：

### 10.1 新增数据源

目前新增源需要同时改代码、文档、配置，而且没有统一 collector 注册机制。

### 10.2 新增推送渠道

推送逻辑目前只面向企业微信 webhook，没有形成通知抽象层。

### 10.3 多主题或多群推送

当前配置结构不适合表达“多个机器人、不同栏目、不同时间策略”。

### 10.4 任务重试与补偿

当前没有明确的任务执行记录模型，无法优雅支持失败重试。

### 建议

- 增加 collector registry
- 增加 notifier 抽象层
- 配置设计支持多 job
- 引入任务运行表

## 11. 过度设计与设计不足

## 11.1 过度设计的部分

- `HotItem.final_score` 与 `time_decay_bonus` 属于旧评分体系残留，当前主逻辑不再依赖，保留会增加理解成本。
- 设计文档体量相对当前项目规模偏重，如果不能持续同步，会变成维护负担。

## 11.2 设计不足的部分

- 配置系统
- 状态存储
- 幂等机制
- 错误恢复
- 部署标准化
- 可观测性
- 安全治理

### 判断

当前系统不是“抽象太多”，而是“业务规则抽象够了，系统工程抽象不够”。

---

## 12. 最值得立即优化的部分

建议按下面顺序处理：

### 第一阶段：先把系统从“危险可用”拉到“稳定可用”

1. 轮换 webhook，清理文档中的敏感信息
2. 修复推送结果判定，只看 HTTP 状态不够，要看业务响应
3. 处理部分成功场景，避免重复推送和状态错乱
4. 给任务加锁，避免重入

### 第二阶段：先补工程契约，再做功能扩展

1. 引入 `pyproject.toml`
2. 明确 dev / runtime 依赖
3. 加 `Makefile`
4. 加 CI
5. 清理 README 与代码不一致项

### 第三阶段：为扩展做边界改造

1. 配置集中加载
2. collector 不再直接读文件
3. 状态迁移到 SQLite
4. 推送记录结构化

## 13. 低成本高收益优化项

这些事项实现成本不高，但收益明显：

- 用 `argparse` 替换手工解析参数
- 把 `safe_collect` 提取为可测试的真实函数
- 增加配置校验
- 增加统一的 `make test`
- 在日志中加入 `run_id`
- 清理未使用或遗留抽象
- 把 RSS feed 真正接入配置
- 部署脚本默认 dry-run

这些事项适合作为短周期优化任务直接落地。

## 14. 中长期重构建议

推荐目标不是“大改重写”，而是逐层收敛边界。

## 14.1 推荐目标架构

```text
app/
  cli.py               # 命令入口

domain/
  models.py            # HotItem, Value Objects
  scoring.py           # 评分规则
  selection.py         # 合并/去重/竞争逻辑

services/
  digest_service.py    # 一次完整任务编排

infra/
  config/
    settings.py
  collectors/
  pushers/
  storage/
    state_store.py
    sqlite_store.py
```

## 14.2 重构原则

- 先抽边界，不先追求复杂框架
- 先把“读配置”“读写状态”“推送事务”从 `main.py` 移出去
- 先保留现有行为，再逐步替换实现

## 14.3 为什么这样改

这样做的核心收益是：

- 业务规则更容易测试
- collector 更容易扩展
- 状态更容易追踪
- CLI、cron、容器、未来 API 都可以复用同一内核

## 15. 部署优化建议

## 15.1 短期建议

- 保留当前 `launchd`/`cron` 方案，但加锁和 dry-run 验证
- 明确日志路径、运行用户、Python 路径
- 把部署脚本改成“安装”和“验证”分离

## 15.2 中期建议

- 增加 Docker 运行方式
- 增加 Linux 兼容部署方式
- 增加环境变量配置模式

## 15.3 长期建议

- 任务调度与应用逻辑解耦
- 使用标准作业系统执行任务
- 形成稳定的发布、回滚、验证流程

## 17. 生产环境建议

如果后续按生产系统维护，最低建议做到：

- secret 外置
- 状态存储持久化
- 分类级幂等
- 推送失败可重试
- 结构化日志
- 失败告警
- 健康检查
- CI 验证
- 标准化部署包

### 生产目标不是“更复杂”

而是让系统具备：

- 出错时知道哪里错
- 重跑时不会重复发
- 迁移时不依赖本机偶然环境
- 接手时不需要猜代码真实行为

## 19. 最终判断

这个项目值得继续优化，不需要推倒重来。

当前最重要的判断不是“代码能不能继续写”，而是：

- 业务抽象已经够用了
- 系统工程边界需要补齐
- 应先修正确性和安全，再做扩展
- 应先补基础工程契约，再做大重构

后续如果继续优化，建议始终遵守这条原则：

> 先消除状态不一致、隐式配置和部署不确定性，再扩展数据源、推送渠道和复杂功能。
