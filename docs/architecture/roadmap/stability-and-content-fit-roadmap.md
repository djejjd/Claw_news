# Claw_news 稳定性与内容命中校准阶段路线图

## 1. 阶段定位

当前项目已经完成以下关键演进：

1. 正式运行链路已统一到 `ai / tool / game`
2. 多类内容已进入正式推送入口
3. GitHub「今日值得看项目」链路已具备：
   - 候选召回
   - 本地排序
   - 曝光惩罚
   - 推荐理由
   - 请求预算控制
   - 失败状态修正

这意味着项目的主要矛盾已经从“功能是否存在”切换为：

1. 服务是否长期稳定
2. 推送内容是否真正符合预期
3. 出问题时是否容易定位和修正

因此，下一阶段不再以扩源或扩功能为主，而是以“稳定性与内容命中校准”为核心目标。

---

## 2. 阶段目标

### 2.1 总目标

让 Claw_news 从“可以运行的聚合工具”进入“可以长期使用、持续校准、问题可判断”的阶段。

### 2.2 子目标

1. 服务稳定运行
   - 采集、评分、推送、部署、健康检查保持稳定
   - 外部源波动不影响主链路可用

2. 内容符合预期
   - AI / 工具 / 游戏 / GitHub 输出整体贴近关注方向
   - 噪音、重复、偏科处于可接受范围

3. 状态可观测、可判断
   - 出现问题时，能够较快判断是源故障、评分偏差、推送失败还是展示问题

---

## 3. 阶段重点

本阶段只关注以下三类问题：

1. 稳定性问题
   - 外部源失败是否会污染状态
   - 超时、重试、optional source 语义是否清晰
   - 健康检查是否能真实反映服务状态
   - 部署后是否容易验证服务是否正常

2. 内容命中问题
   - 最终推送是否值得看
   - 三类内容是否平衡
   - GitHub 项目是否有明显噪音
   - 重复控制是否合理
   - 是否存在“能抓到，但不值得推”的情况

3. 归因效率问题
   - 出现异常时，能否快速定位问题归属
   - 是否能区分：
     - 源问题
     - 排序/评分问题
     - 推送问题
     - 展示问题

---

## 4. 阶段完成标准

当以下条件基本满足时，可认为本阶段完成：

1. 服务连续运行稳定，定时任务、推送、健康检查无明显异常
2. 外部源偶发失败不会导致整体服务不可用，健康状态能真实反映失败情况
3. 每日推送内容整体符合预期，AI / 工具 / 游戏结构基本合理
4. GitHub 项目推荐噪音明显可控，重复处于可接受范围，不会长期霸榜刷屏
5. 最终推送结果总体“值得看”，而不是只是“能抓到很多内容”
6. 出现异常时，可以较快定位问题归因并采取处理

---

## 5. 本阶段不做什么

为避免目标漂移，本阶段明确不做以下内容：

1. 不以继续扩源为主要目标
2. 不做大规模架构重写
3. 不把 GitHub 项目并入主新闻 candidate 主链路
4. 不做复杂的用户反馈学习或个性化训练
5. 不为了增加指标而堆更多展示内容
6. 不在没有明确收益的情况下新增线上基础设施

如果某项修改主要服务于“新增功能”而不是“稳定或校准”，不应纳入本阶段。

---

## 6. 执行原则

后续开发必须遵守以下原则：

1. 先稳后扩
   - 优先修正会影响线上稳定性、状态真实性和内容判断的问题

2. 小步可验证
   - 每一轮修改都应有明确验证方式
   - 不接受“改了很多，但无法证明结果更好”的改动

3. 范围收敛
   - 每个任务只处理一个清晰问题族
   - 不把无关优化混入同一轮实现

4. 结果可审查
   - 每个任务完成时都必须提供：
     - 改动范围
     - 验证结果
     - 残留风险
     - 明确不做项

---

## 7. 阶段任务拆分

本阶段拆分为 4 个任务组，建议按顺序执行。

### Task 1：状态与验收基线设计

#### 目标

先把“系统当前应该如何表达真实状态、后续如何判断是否达标”定义清楚，形成后续治理与校准的统一基线。

#### 要解决的问题

1. `/health` 应该表达哪些状态，才能支撑“服务稳定运行”的判断
2. ingest / publish / source 的状态语义应该如何区分
3. 哪些状态属于“本轮态”，哪些属于“历史态”，哪些属于“验收态”
4. 为后续内容命中校准，哪些关键字段必须持久化
5. 当前哪些外部源应被视为稳定源、可选源、降级源或问题源

#### 建议产出

1. 状态模型说明
   - ingest status
   - publish status
   - source status
   - overall health status

2. `/health` 返回模型
   - 服务进程状态
   - scheduler 状态
   - ingest 新鲜度
   - publish 最近结果
   - source 级别失败/降级信息

3. source 状态语义表
   - success
   - failed
   - skipped
   - degraded
   - optional failure

4. 状态作用域定义
   - 本轮状态
   - 时间窗口状态
   - 历史记录

5. 校准证据字段清单
   - 正文候选/最终入选需要落盘哪些字段
   - GitHub 最终入选需要落盘哪些字段

6. 异常类型清单
   - timeout
   - HTTP 4xx / 5xx
   - 解析失败
   - push 失败
   - 状态污染

#### 完成标准

1. 可以明确回答“系统当前应该如何表达真实状态”
2. 可以明确回答“哪些状态要写到 `/health`，哪些不该混在一起”
3. 可以明确回答“后续内容命中校准依赖哪些持久化证据”
4. 为 Task 2 提供明确实现目标，不再靠口头补充边界

#### 注意事项

1. 本任务的重点是定义，不是立即把所有逻辑实现完
2. 如需改代码，只应限于为 Task 2 铺路的最小结构调整
3. 不要在本任务内直接做内容排序优化
4. 不要把“状态模型设计”简化成只写几条文字，要能直接指导实现

---

### Task 2：状态真实性与发布一致性治理

#### 目标

修正当前会直接影响状态真实性、发布一致性和后续判断依据的问题，为后续内容命中校准提供可信地基。

#### 要解决的问题

1. `/health` 当前恒定绿灯，不能真实反映服务状态
2. RSS 部分 feed 失败会被吞掉，导致 `rss` 整体被误记成功
3. `source_failures` 会跨轮次累积，污染后续 publish 视图
4. push 成功后的多处状态写入缺少足够一致性，存在部分提交污染
5. `POST /run/news` 返回值中的 `pushed` 语义不真实
6. publish 失败没有 durable 状态，后续难以排查
7. 内容命中校准所需关键证据没有完整落盘

#### 重点方向

1. 修正 `/health`
   - 不再恒定返回 `healthy`
   - 至少能体现：
     - ingest 是否陈旧
     - 最近 publish 是否失败
     - 是否存在 failed / degraded source
     - scheduler 是否处于可运行状态

2. 修正 RSS 部分失败语义
   - 单个 RSS feed 失败不能继续伪装成整个 `rss` 成功
   - 要能表达“部分成功 / degraded”

3. 修正 `source_failures` 污染
   - 区分本轮失败与历史失败
   - 不允许一次旧失败持续污染后续窗口发布结果

4. 修正 push 后状态一致性
   - push 成功后的 exposure / pushed URLs / published keys / digest / metrics 写入需要具备清晰一致性语义
   - 至少要做到：
     - 失败能被看见
     - 部分提交能被识别
     - 返回值不能继续伪装为 `ok`

5. 修正手动触发返回语义
   - `pushed` 必须只代表主推送是否成功
   - 异常通知是否发出不能混入同一字段

6. 补齐 durable publish 状态
   - 失败不能只在运行时短暂返回
   - 后续排查必须能看到最近 publish 的真实结果

7. 补齐校准证据落盘
   - 正文最终入选至少应可回看：
     - `source`
     - `display_category`
     - `topic_label`
     - `topic_confidence`
     - `final_score`
   - GitHub 最终入选至少应可回看：
     - `final_score`
     - `activity`
     - `popularity`
     - `relevance`
     - `penalty`
     - `recommendation`
     - `matched_topics`
     - `matched_keywords`

#### 完成标准

1. `/health` 不再只是“进程活着”的伪健康，而能支撑基本判断
2. RSS、GitHub、其他 source 的失败/跳过/降级语义更真实
3. publish 相关状态不再出现明显污染或伪成功
4. 手动触发返回值可用于部署后真实验证
5. 后续内容校准所需关键证据已经落盘，能够支持回看和归因

#### 注意事项

1. 本任务可以一次处理掉以上问题，不必强行拆散到后续任务
2. 不要把这一任务扩展成全面监控或大规模基础设施改造
3. 不要求本轮一定引入数据库；如果继续使用文件存储，也必须把状态语义设计清楚
4. 不要在本任务中顺手做 GitHub 排序优化或扩源

---

### Task 3：内容命中校准

#### 目标

让最终推送更接近“值得看”，而不是只是“抓到内容”。

#### 要解决的问题

1. 三类正文内容是否平衡
2. GitHub 候选是否仍存在明显噪音
3. 重复控制是否合理
4. source/topic 权重是否造成偏科
5. 是否存在明显灌水或无关项进入 top10

#### 重点方向

1. GitHub 抗噪
   - 继续优化 query 和 relevance 规则
   - 加负向过滤或硬门槛

2. 正文分布校准
   - AI / 工具 / 游戏是否长期失衡
   - 是否需要对 source weight / topic weight 做小幅修正

3. 重复控制
   - 不追求“绝不重复”
   - 重点是避免无意义重复和长期霸榜

4. 结果质量判断
   - 不是只看抓取数，而是看最终推送是否有价值

#### 完成标准

1. GitHub 噪音明显下降
2. 最终 top10 的主观可读性提升
3. 三类内容结构更接近预期
4. 重复虽存在但合理，不令人厌烦

#### 注意事项

1. 不要在本任务中继续大规模扩源
2. 不要做复杂的反馈学习系统
3. 只做能直接改善命中质量的调整

---

### Task 4：观测与验收

#### 目标

建立后续判断“系统好不好使”的观察方法和验收口径。

#### 要解决的问题

1. 以后如何判断项目是否达标
2. 每天或每次发版后应该看什么
3. 发现问题后如何归因

#### 建议产出

1. 日常观察项清单
   - 健康状态
   - 成功源/失败源
   - 正文结构
   - GitHub 噪音
   - 重复情况

2. 人工验收模板
   - 今日推送是否值得看
   - 哪些条目偏离预期
   - 偏差属于：
     - 源问题
     - 评分问题
     - 展示问题
     - 推送问题

3. 简单验证流程
   - 发版后看什么
   - 手动触发时看什么
   - 出现异常时先查什么

#### 完成标准

1. 后续不需要每次从零解释“这个系统为什么看起来不对”
2. 能快速判断本轮问题属于稳定性还是命中质量
3. 为后续小步迭代提供持续反馈依据

#### 注意事项

1. 这一任务重点是沉淀判断方式，不是搭大而全的平台
2. 优先使用现有状态和日志，不要为了观测引入重型系统

---

## 8. 任务执行顺序

建议严格按以下顺序推进：

1. Task 1：稳定性基线梳理
2. Task 2：稳定性治理
3. Task 3：内容命中校准
4. Task 4：观测与验收

原因如下：

1. 不先看清现状，就容易把“感受上的问题”误判成“代码必须重写”
2. 稳定性不先收住，后面的内容校准结果不可信
3. 没有验收口径，后续每一轮优化都难以判断是否真的变好

### 8.1 当前启动范围

基于当前代码审查结果，本阶段**现在只启动前两个任务**：

1. Task 1：稳定性基线梳理
2. Task 2：稳定性治理

当前**不直接启动**以下任务：

1. Task 3：内容命中校准
2. Task 4：观测与验收

原因不是这两个任务不重要，而是当前系统还存在会直接影响判断质量的问题：

1. `/health` 真实性不足，不能代表“服务稳定运行”
2. RSS 部分失败语义不真实
3. `source_failures` 存在跨轮次污染
4. push 成功后的状态写入一致性不足
5. publish 失败缺少 durable 状态
6. 内容校准所需的关键证据没有完整落盘

在这些问题修正前，过早进入内容命中校准，会出现“能调整结果，但很难证明为什么变好或变差”的情况。

### 8.2 Task 3 / Task 4 的启动前提

只有当前两个任务完成，并且以下条件基本满足后，才启动 Task 3 和 Task 4：

1. `/health` 能较真实地反映服务状态，而不是恒定绿灯
2. ingest / publish 的成功、失败、跳过语义清晰且不互相污染
3. push 后的关键状态写入具备足够一致性
4. publish 失败能够被后续排查，而不是只在运行时短暂返回
5. 用于内容判断的关键字段已经持久化，能支持回看与归因

如果以上前提未满足，Task 3 和 Task 4 不应提前开始。

## 9. 每个任务的统一交付要求

后续无论由谁实现，每个任务完成时都必须提交以下内容，避免审查时信息不对称：

1. 改动目标
   - 这轮要解决什么问题
   - 不解决什么问题

2. 改动文件清单
   - 修改了哪些文件
   - 每个文件的职责变化是什么

3. 验证结果
   - 跑了哪些测试
   - 做了哪些手动验证
   - 线上或本地验证到什么程度

4. 残留风险
   - 只列本轮直接相关风险
   - 不扩写无关 roadmap

5. 不做项
   - 明确说明这轮故意没有做什么

---

## 10. 审查标准

后续所有任务审核都以以下标准为准：

1. 是否服务于本阶段目标
   - 稳定性
   - 内容命中
   - 可观测性与归因效率

2. 是否引入不必要复杂度
   - 不接受为了“看起来完整”而堆复杂结构

3. 是否可验证
   - 没有测试或验证证据的“优化”不视为完成

4. 是否控制范围
   - 不接受在同一轮里混入无关功能扩展

5. 是否保留正确语义
   - 失败不能伪装成功
   - 状态不能被污染
   - 命中质量调整不能破坏主链路稳定

---

## 11. 推荐执行方式

本阶段推荐采用以下协作方式：

1. 每次只推进一个任务组
2. 一个任务组内允许拆小任务并行，但必须共享同一边界和验收标准
3. 每轮先交付实现，再做集中审核
4. 审核不过的，先补回归和语义问题，再谈后续优化

这样可以保证：

1. 节奏稳定
2. 变更边界清晰
3. 不会在“扩题”和“返工”之间反复横跳

---

## 12. 一句话总结

这一阶段的目标不是让 Claw_news“功能更多”，而是让它“更稳定、更可信、更符合预期、更容易判断问题”。
