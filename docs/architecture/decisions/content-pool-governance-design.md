# 内容候选池与选材治理设计

## 1. 文档定位

本文定义 Claw_news 正文候选池、内容相关性过滤、跨日补位和最终选材的统一契约，同时定义后续由多个 AI 执行开发时的任务包与交付规则。

本文只覆盖 `ai / tool / game` 正文链路，不修改 GitHub 项目推荐链路。

## 2. 背景与问题

当前系统每 30 分钟采集一次内容，每天推送一次 Top 10。高频综合源会在一天内持续产生新文章，而低频垂直源和深度源通常只更新少量文章，导致候选分布和最终推送同时出现偏科。

当前主要问题如下：

1. 高频源依靠数量优势占据候选池大多数。
2. 当前 `final_score` 主要由统一源权重和时间项构成，缺少足够的来源质量差异。
3. 单源多样性惩罚只能缓解最终霸榜，不能过滤不相关内容。
4. 当前发布窗口只读取当天内容，无法在某类内容不足时使用前几天仍有价值的文章。
5. `seen_keys`、候选存储、推送资格和已发布状态的职责容易混淆。
6. 现有 `TopicClassifier` 主要面向 AI 话题，不能直接作为工具和游戏分类的完整相关性判断器。

## 3. 设计目标

1. 原始候选池继续完整接收去重后的采集内容，不因推送策略丢失校准证据。
2. 推送前过滤明显不相关内容，避免高频综合源凭数量优势刷屏。
3. 最终 Top 10 优先满足 AI、工具、游戏三个分类的最低数量。
4. 当天某分类不足时，可以按来源类型从前几天未发布的高分文章中补位。
5. 低频精品源获得更长的内容有效期，但不能绕过相关性判断。
6. 保留唯一的排名分数 `final_score`，不引入第二套候选准入总分。
7. 所有过滤、补位和入选结果可解释、可测试、可回放。
8. 第一阶段人工维护源策略，同时为未来自动生成分级建议预留数据和接口。

## 4. 非目标

1. 不实现源类型自动迁移或自动修改线上配置。
2. 不引入数据库、消息队列或新的线上基础设施。
3. 不使用 LLM 对每篇候选执行相关性判断。
4. 不设置简单粗暴的最终单源硬上限。
5. 不为凑满 10 条而降低相关性门槛。
6. 不重写现有采集、摘要和推送主链路。
7. 不调整 GitHub 推荐排序。

## 5. 核心概念与状态边界

### 5.1 原始候选池

原始候选池保存所有通过基础字段校验且完成 canonical key 去重的文章。基础字段校验仍包括标题、URL、受支持分类和最小摘要长度。

文章不需要先获得相关性分数才能写入原始候选池。

### 5.2 本次可选集合

本次可选集合是发布时从原始候选池动态计算的逻辑视图。一篇文章必须同时满足以下条件：

1. 未发布；
2. 未超过所属源的有效期；
3. 通过本次相关性过滤；
4. 属于 `ai / tool / game`；
5. 发布时间或回退时间落在最大回看范围内。

本次可选集合不要求建立新的数据库表或独立存储文件。

### 5.3 最终入选集合

最终入选集合是本次从可选集合中选出的最多 10 条文章。未入选文章只要未发布且未过期，后续仍可参与跨日补位。

### 5.4 状态职责

- `seen_keys`：仅表示文章曾被采集，防止高频任务重复写入。
- 原始候选记录：保存采集结果和后续校准证据。
- `pushed_urls / published_keys`：表示文章已经成功发布，长期阻止重复推送。
- 来源有效期：决定未发布文章当前是否仍有推送资格。

不得通过清空 `seen_keys` 解决来源失衡。清空只会让 RSS 中的旧条目被重复采集。

## 6. 来源策略

### 6.1 人工分级

第一阶段为每个来源显式配置策略：

| tier | 默认有效期 | 过滤强度 | 典型用途 |
|---|---:|---|---|
| `fast_news` | 24 小时 | 严格 | 高频综合资讯、榜单 |
| `vertical` | 48 小时 | 标准 | 垂直领域资讯 |
| `deep` | 72 小时 | 宽松但保留相关性底线 | 低频深度精品内容 |

`tier` 只决定有效期和过滤策略，不直接等于来源质量。

### 6.2 来源质量权重

每个源单独配置 `quality_weight`，用于表达长期来源质量。它与 `tier` 独立：

- `tier` 决定文章可以保留多久；
- `quality_weight` 决定相同新鲜度下哪个源更优先。

缺少显式配置的新源使用以下保守默认值：

```yaml
tier: vertical
retention_hours: 48
quality_weight: 3.0
filter_profile: standard
```

|---|---|---:|
| `ithome` | `fast_news` | 2.0 |
| `taptap` | `fast_news` | 2.5 |
| `eurogamer` | `fast_news` | 3.0 |
| `qbitai` | `vertical` | 3.5 |
| `leiphone` | `vertical` | 3.5 |

初始值是可校准基线，不是永久结论。

### 6.4 配置示例

来源策略应随 feed 配置维护，避免排序器维护固定来源名单：

```yaml
- url: https://www.ithome.com/rss/
  source: ithome
  tier: fast_news
  retention_hours: 24
  quality_weight: 2.0
  filter_profile: strict
```

## 7. 时间与存储规则

### 7.1 时间基准

有效期优先使用 RSS 提供的完整发布时间。RSS 缺少可靠发布时间时，使用首次抓取时间作为回退。

“今日”按服务配置时区 `Asia/Shanghai` 的自然日判断；来源有效期按触发时刻向前滚动 24、48、72 小时判断。二者不得混用。

当前只保存 `yyyy-mm-dd` 的行为无法精确执行 24、48、72 小时有效期，实施时必须保留可解析的完整 ISO 时间。旧候选缺少完整时间时允许回退到已有日期或首次抓取时间，并记录使用的时间来源。

### 7.2 原始候选保留

原始候选记录保留 7 天：

- 在来源有效期内：可以参与本次推送；
- 超过来源有效期但未超过 7 天：只用于审计和校准；
- 超过 7 天：由清理任务删除。

7 天存储保留与最长 72 小时推送有效期是两个独立概念。

### 7.3 指标保留

每日聚合的来源新增率、相关性通过率、入选率和拒绝原因可以保留 30 至 90 天，为未来自动分级建议提供依据。第一阶段只记录证据，不自动修改来源策略。

## 8. 相关性过滤

### 8.1 执行位置

相关性过滤在发布读取原始候选后、最终评分和选材前执行。它不阻止不重复文章写入原始候选池。

### 8.2 判断流程

1. 命中明确排除规则：拒绝参与本次推送；
2. 命中明确正向领域规则：接受；
3. 规则无法明确判断：交给分类相关性规则复核；
4. 复核仍不满足门槛：拒绝参与本次推送。

规则需要同时查看标题和摘要，并按 AI、工具、游戏分别维护。现有 `TopicClassifier` 可以复用 AI 话题结果，但必须补充独立的跨分类相关性组件，不能把 AI fallback 结果当成工具和游戏的有效判定。

### 8.3 来源过滤配置

- `strict`：用于高频综合源，要求更强的领域证据；
- `standard`：用于垂直资讯源；
- `lenient`：用于可信深度源，减少标题关键词不足造成的误杀，但仍必须满足分类相关性底线。

`lenient` 不代表无条件接受。

三档 profile 使用统一的冲突优先级和不同置信度门槛：

1. 排除规则优先于正向规则；同一文章同时命中时默认拒绝，并记录冲突原因；
2. 明确命中分类正向规则时直接接受；
3. 其余文章进入跨分类相关性复核；
4. `strict` 要求复核置信度不低于 `0.7`；
5. `standard` 要求复核置信度不低于 `0.5`；
6. `lenient` 要求复核置信度不低于 `0.3`。

跨分类相关性组件必须为 AI、工具、游戏返回同一量纲的 `0.0` 至 `1.0` 置信度。具体正向词、排除词和主题规则属于可配置内容，必须在实施任务包中列出初始配置及对应测试样本，不得由开发 AI 临时发挥。

### 8.4 解释字段

相关性判断至少输出：

- `relevance_status`：`accepted` 或 `rejected`；
- `relevance_reason`：命中的正向规则、排除规则或复核结论；
- `topic`；
- `topic_confidence`。

是否将以上字段回写候选记录或写入独立审计记录，由实施计划在不破坏追加型存储语义的前提下确定；最终发布证据必须能够回看。

## 9. 唯一排名公式

系统只保留一套最终排名分数：

```text
final_score = source_quality_weight + freshness_score
```

相关性只决定能否参与本次推送，不再额外叠加一套 `admission_score`。来源类型也不直接进入排名公式。

`freshness_score` 必须使用小时级文章年龄，并保证：

1. 今天的合格内容自然优先；
2. 旧内容随时间单调衰减；
3. 同等来源质量下，较新内容分数更高；
4. 缺少可靠发布时间时的回退行为可测试、可解释。

第一阶段使用以下固定分段，不允许开发 AI 在实现时自行调参：

| 文章年龄 | freshness_score |
|---|---:|
| `0h <= age <= 24h` | 3.0 |
| `24h < age <= 48h` | 1.5 |
| `48h < age <= 72h` | 0.5 |
| 超过所属源有效期 | 不可参与 |

未来调整分段必须基于回放证据，并先修改本文或对应决策文档。

## 10. 最终选材算法

最终数量上限为 10，分类最低目标为：

- AI：3 条；
- 工具：2 条；
- 游戏：2 条。

### 10.1 阶段一：今日分类保底

只从今天的合格内容中，按 `final_score` 和单源多样性有效分依次选择 AI 3、工具 2、游戏 2。

### 10.2 阶段二：跨日分类补位

当天某分类不足时，从同分类历史候选中补足最低目标。历史候选必须：

1. 未发布；
2. 通过相关性过滤；
3. 未超过所属源有效期；
4. 位于最大 72 小时读取范围内。

跨日候选按 `final_score` 和单源多样性有效分选择。跨日内容只用于补足分类最低目标，不参与自由竞争名额。

例如当天只有 1 条合格 AI，系统可以使用 48 小时内未发布的量子位文章和 72 小时内未发布的美团技术文章，将 AI 补到 3 条。

### 10.3 阶段三：今日自由竞争

分类最低目标处理完成后，剩余名额只从当天未入选的合格文章中竞争，直到达到 10 条或今日可选内容耗尽。

若跨日补位后仍无法满足分类最低目标，系统允许最终少于 10 条，不降低相关性门槛，不使用过期内容凑数。

### 10.4 单源多样性

现有递增单源惩罚继续使用，并在三个选材阶段中累计同一份 `source_counts`。不得在进入下一阶段时重置来源计数。

第一阶段固定沿用当前惩罚：同源已入选 0、1、2、3、4 条及以上时，对下一候选分别应用 `0.0 / -1.0 / -2.0 / -3.5 / -5.0`。选材有效分为：

```text
selection_score = final_score + source_diversity_penalty
```

`selection_score` 只用于逐步选材，不替代持久化的 `final_score`。

第一阶段不增加最终单源硬上限。真正重要的连续报道仍可在分数足够高时入选，但同一源每增加一条，下一条需要克服更高惩罚。

最终展示顺序可以按原始 `final_score` 排序，但选材过程和审计证据必须记录应用多样性惩罚后的有效分，避免出现“展示分数无法解释入选顺序”的问题。

## 11. 读取与数据流变化

当前发布读取窗口为当天 `00:00` 到触发时刻。新流程应：

1. 读取触发时刻向前最多 72 小时涉及的候选文件；
2. 过滤已发布 URL 和 canonical key；
3. 按每篇文章所属源策略应用 24、48、72 小时有效期；
4. 执行相关性判断；
5. 计算唯一 `final_score`；
6. 执行三阶段选材；
7. 将选材原因和评分证据写入现有 digest 或明确的审计载体。

读取范围扩大不改变原始候选的 7 天物理保留策略。

## 12. 错误与降级语义

1. 来源策略配置字段非法：配置加载失败并明确指出来源与字段，不静默接受错误值。
2. 新源缺少策略：使用保守默认值并记录告警。
3. 文章缺少完整发布时间：使用首次抓取时间；仍不可用时不得作为跨日补位候选。
4. 相关性组件异常：明确命中正向规则的文章仍可参与；无法判断的文章本轮拒绝参与，避免无关内容刷屏。
5. 历史候选读取失败：本轮降级为只使用今日候选，并记录 publish 错误或降级原因。
6. 某分类不足：允许少发，不降低过滤门槛。
7. 审计证据写入失败：沿用 durable publish 状态的降级语义，不伪装为完整成功。

## 13. 可观测性与未来扩展

第一阶段至少按来源记录：

- 原始抓取数；
- 去重后新增数；
- 相关性接受数与拒绝数；
- 各拒绝原因计数；
- 今日入选数；
- 跨日补位数；
- 最终入选率。

未来自动化只允许先生成来源分级建议，不直接修改线上配置。至少积累 14 至 30 天有效数据后，才评估有限自动调整；任何自动调整都必须具备冷却期、变更记录和回滚能力。

## 14. 验收标准

1. `ithome` 即使占原始候选池大多数，也不会仅凭数量优势占据最终推送多数。
2. 明显不相关的高频综合资讯不能进入最终选材。
3. 今日合格候选充足时，优先满足 AI 3、工具 2、游戏 2。
4. 今日某分类不足时，可从对应分类仍有效的 48 或 72 小时精品内容中补位。
5. 跨日文章只补分类最低目标，不参加今日自由竞争。
6. 已发布、过期或相关性不合格文章不会入选。
7. 单源多样性惩罚跨三个阶段累计生效。
8. 原始候选保留 7 天，超过来源有效期的记录仍可审计但不可推送。
9. 完整发布时间存在时按小时计算；缺失时按契约回退。
10. 每篇过滤、补位和入选文章都能解释原因。
11. 旧配置可以按明确默认值兼容，不要求一次性补齐所有自定义源。
12. 历史样本回放能够对比修改前后的来源分布、分类分布和人工相关性评价。

## 15. 测试策略

### 15.1 单元测试

1. 来源策略解析、默认值和非法配置；
2. 24、48、72 小时有效期边界；
3. 完整发布时间与首次抓取时间回退；
4. 三类正向、排除和模糊相关性规则；
5. 今日分类最低目标；
6. 跨日同分类补位；
7. 跨日候选不得进入自由竞争；
8. 单源计数跨阶段累计；
9. 最终不足 10 条时不凑数；
10. 已发布和过期过滤。

### 15.2 集成测试

构造高频源占绝对多数、慢源少量精品、某分类当天不足的混合候选集，验证完整读取、过滤、评分、补位、选材和证据落盘。

### 15.3 历史样本回放

使用已落盘候选做离线回放，至少输出：

- 修改前后来源分布；
- 修改前后分类分布；
- 今日与跨日内容数量；
- 相关性拒绝原因；
- 人工标记的明显无关条目数量。

回放不得触发真实推送或修改生产发布状态。

## 16. 多 AI 协作契约

### 16.1 规则文件职责

仓库应新增根目录 `AGENTS.md` 作为所有开发 AI 的公共入口。共享规则只维护一份，避免与 `CLAUDE.md` 漂移。

- `AGENTS.md`：项目目标、文档入口、开发门禁、验证命令、任务执行和交付规则；
- `CLAUDE.md`：要求先完整阅读 `AGENTS.md`，只保留 Claude 专属适配内容；
- 当前任务包：约束本次允许做什么；
- 本设计：约束功能语义和验收边界。

指令优先级为：

```text
用户当前指令
> 当前任务包
> 已批准设计文档
> AGENTS.md
> CLAUDE.md 中的工具专属补充
> 历史代码注释与归档文档
```

### 16.2 强制停止条件

每个任务包必须包含：

> 若任务包、批准设计、现有代码三者存在无法兼容的冲突，立即停止实现，报告冲突位置、影响和可选处理方案；不得自行扩大范围、修改契约或用静默兼容掩盖冲突。

### 16.3 任务包结构

每个实施 Task 必须包含：

1. 背景；
2. 目标；
3. 前置依赖；
4. 输入与输出契约；
5. 修改范围；
6. 禁止事项；
7. 执行要求；
8. 实施步骤；
9. 验收标准；
10. 检查命令；
11. 交付前自检；
12. 交付格式。

每个 Task 还必须标明：任务编号、依赖任务、允许并行任务、预计修改文件、不得修改文件和完成状态。

### 16.4 交付前自检

开发 AI 在交付前必须逐项确认：

1. 已完整阅读 `AGENTS.md`、任务包和任务指定设计章节；
2. 只修改了允许范围；
3. 没有擅自改变接口、状态语义、默认值或评分常量；
4. 新行为具备正常、边界、降级和失败路径测试；
5. 没有静默 fallback、吞异常或伪成功；
6. 可配置规则没有被重新硬编码到业务逻辑；
7. 旧配置兼容行为符合任务包；
8. 示例配置和中文文档已同步；
9. 已运行任务包指定的精确测试；
10. 已运行要求的完整测试、lint 和格式检查；
11. 已检查 diff 中的意外修改、调试代码、密钥和机器绝对路径；
12. 每条验收标准都有对应证据；
13. 未验证内容明确标记，未被描述为已完成。

### 16.5 固定交付格式

每个开发 AI 必须按以下结构交付：

1. 完成内容；
2. 实际修改文件；
3. 与设计契约的逐条对应；
4. 执行的检查命令及结果；
5. 新增或修改的测试；
6. 未完成项；
7. 残留风险；
8. 发现的设计冲突；
9. 范围外改动声明；
10. 建议主审核 AI 重点检查的位置。

## 17. 建议实施任务拆分

后续实施计划至少拆分为以下任务包，具体文件边界和检查命令必须在正式计划中展开：

1. Task 1：建立 `AGENTS.md` 与协作交付契约；
2. Task 2：扩展来源策略配置与完整发布时间数据契约；
3. Task 3：实现候选 7 天保留、72 小时读取和按源有效期过滤；
4. Task 4：实现推送前跨分类相关性过滤；
5. Task 5：实现分类保底、跨日补位和今日自由竞争；
6. Task 6：接入来源质量权重与跨阶段单源惩罚；
7. Task 7：补齐审计证据、示例配置和离线回放；
8. Task 8：全链路回归、历史样本回放与验收报告。

Task 2 是后续数据契约基础。Task 3 和 Task 4 可以在 Task 2 完成后并行。Task 5 和 Task 6 必须基于读取、过滤和数据字段完成。Task 7、Task 8 最后收口。

## 18. 实施边界

正式实施前必须另行编写详细实施计划。计划不得把本文的建议任务标题直接当作可执行任务，必须为每个 Task 补齐第 16.3 节规定的十二项内容。

实施中的新想法默认进入后续项，不得由开发 AI 在当前 Task 中顺手扩展。任何需要改变分类最低目标、来源有效期、唯一评分公式、相关性门槛语义或跨日补位边界的调整，都必须先修改并重新批准本文。
