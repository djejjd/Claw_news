## 3. 必须修改

## 3.1 部分失败仍以整体成功结束

### 涉及位置

- `main.py:77-122`
- `main.py:174-184`

### 当前行为

`run_push_sequence()` 已经能在单个 category 推送异常时：

- 记录错误日志
- `continue` 到下一个 category
- 保留已成功 category 的状态提交

这是正确的第一步。

但 `main()` 当前无论中间失败多少次，最后都会执行：

```python
cleanup_old_digests()
logger.info("推送完成")
```

并以成功路径结束。

### 为什么这是问题

从生产视角，任务结果至少有三种：

- 全成功
- 部分失败
- 全失败

当前实现把“部分失败”也当作“成功完成”处理，会导致：

- 调度器无法感知任务异常
- CI / cron / launchd 无法基于退出码判断是否告警
- 日志结论与真实外部状态不一致
- 运维和排障会误判

### 风险后果

- 失败 silently pass
- 监控缺失
- 推送未全量成功但无人知晓
- 后续补偿逻辑无法基于可靠状态触发

### 推荐修改

让 `run_push_sequence()` 返回结构化结果，例如：

```python
{
    "success_categories": [...],
    "failed_categories": [...],
}
```

或定义专门的 dataclass。

然后在 `main()` 中：

- 全成功：`logger.info("推送完成")`，退出码 `0`
- 部分失败：`logger.error("部分推送失败")`，退出非 `0`
- 全失败：`logger.error("推送失败")`，退出非 `0`

### 是否阻塞合并

**是，阻塞合并。**

## 3.2 README 的配置契约与实现不一致

### 涉及位置

- `README.md:39-44`
- `infra/config/settings.py:27-30`

### 当前文档表述

README 当前写法是：

```bash
# 2. 配置 webhook（二选一）
#    方式 A: 环境变量（推荐）
export PUSHER_WECOM_WEBHOOK=...
#    方式 B: YAML 配置
cp config.example.yaml config.yaml
```

这会让读者自然理解成：

- 只配环境变量也能启动
- 或只配 YAML 也能启动

### 当前真实实现

`Settings.load()` 先检查 `config.yaml` 是否存在：

```python
if not config_path.exists():
    raise ValueError(...)
```

也就是说：

- 即使已经设置了 `PUSHER_WECOM_WEBHOOK`
- 如果 `config.yaml` 不存在
- 仍然会直接失败

### 为什么这是问题

这是典型的 **部署契约不一致**：

- 文档承诺环境变量可单独使用
- 代码实际要求 `config.yaml` 必须存在

这会直接导致：

- 新机器部署踩坑
- 使用者误判
- 运维同学按照 README 操作却启动失败

### 推荐修改

二选一：

#### 方案 A：改代码

支持 env-only 启动：

- `config.yaml` 可缺失
- 若环境变量满足必需配置，则允许启动

#### 方案 B：改文档

明确真实契约是：

- `config.yaml` 仍然必须存在
- 环境变量只用于覆盖其中的 webhook 字段

### 是否阻塞合并

**是，阻塞合并。**

因为这已经是用户可见的错误契约，不是单纯文案优化。
