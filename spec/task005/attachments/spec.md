# Claw_news M6 Task Contract Spec

## Immutable Rule

本文件是当前任务的唯一验收契约。
自确认后：

1. Review 只能基于本文件判断是否通过。
2. Review 不允许提出新的范围外阻塞项。
3. 新发现但不属于本文件范围的问题，必须记录到 `next_task`，不得阻塞当前任务。
4. 阻塞项必须能映射到本文件的“实现要求”或“验收标准”。
5. 禁止使用“建议优化”“最好”“可以考虑”作为阻塞理由。

---

## 任务目标

为当前仓库建立一套清晰、统一、可持续维护的任务文档目录规范，使任何人都能快速判断：

1. 一个 task 的唯一正式契约是什么
2. 一个 task 对应的设计文档是什么
3. 一个 task 对应的实施计划是什么
4. 一个 task 的 review 结论和完成归档在哪里
5. 本轮 task 的目标是什么

---

## 修改范围

本轮允许修改以下内容：

1. `spec/` 目录下与任务文档治理相关的文档
2. `docs/superpowers/specs/` 与 `docs/superpowers/plans/` 中与任务文档迁移方案相关的设计文档
3. 新增目录治理说明文档

本轮默认以“规范设计与迁移方案”作为目标，不强制执行大规模文件迁移。

---

## 禁止修改范围

本轮禁止：

1. 修改业务代码
2. 修改现有 task 的业务验收逻辑
3. 借目录整改之名重写历史 task 内容
4. 删除现有 task 文档原件而不提供迁移方案

---

## 实现要求

### 1. 目录规范契约

1. 必须定义统一的 task 目录结构。
2. 每个 task 必须有固定命名的正式文档入口。
3. 目录规范必须明确：
   - `contract.md`
   - `design.md`
   - `plan.md`
   - `review.md`
   - `done.md`
4. 必须定义历史附件或草稿的放置位置，例如 `attachments/` 或 `archive/`。

### 2. 目标可见性契约

1. 每个 task 的 `contract.md` 必须单独包含 `## 任务目标`。
2. 每个 task 的 `design.md` 必须单独包含 `## 本轮目标`。
3. 每个 task 的 `plan.md` 开头必须再次重申本轮目标。
4. 必须在总索引中为每个 task 提供一句话目标摘要。

### 3. 总索引契约

1. 必须为 `spec/` 目录设计一份总索引文档，例如 `spec/README.md`。
2. 总索引必须至少包含：
   - task 编号
   - 状态
   - 一句话目标
   - 正式文档入口

### 4. 迁移契约

1. 必须给出现有文档到新结构的映射方案。
2. 必须至少覆盖 `task001` 到 `task005` 的迁移建议。
3. 必须区分：
   - 立即迁移
   - 保留历史
   - 仅加索引，不移动原文件

### 5. 落地策略契约

1. 必须明确这是一次“目录治理任务”，不是业务任务。
2. 必须给出推荐执行顺序。
3. 必须明确迁移后的 review 规则：
   - review 只对 `contract.md` 负责

---

## 验收标准

### A. 目录规范清晰

1. 已定义统一的 task 目录结构。
2. 已定义每类正式文档的固定命名。

### B. 目标表达清晰

1. 已明确规定每个 task 都要单独写本轮目标。
2. 已明确规定目标在 `contract.md`、`design.md`、`plan.md` 中的固定位置。

### C. 索引方案成立

1. 已设计 task 总索引结构。
2. 已说明总索引最少包含哪些字段。

### D. 迁移方案可执行

1. 已给出 `task001` 到 `task005` 的迁移映射思路。
2. 已明确迁移顺序与策略。
